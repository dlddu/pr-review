# 테스트 문서: PR 메타데이터 추출

## 검증 대상 AC
- AC1-1: 작업 유형 분류 (PRD-1)
- AC1-2: PR 작성자 식별 — 봇이면 실제 유저 추정 (PRD-1)
- AC1-3: 저장소 식별 (PRD-1)
- AC1-4: 메타데이터의 지식 키 정규화 (PRD-1)

## 테스트 시나리오

### 시나리오 1: 작업 유형이 분명한 PR 분류
- **사전 조건**: 사용자 정의 작업 유형 집합이 존재(기능/버그/리팩토링/문서 등). "버그 수정" 성격의 PR 입력.
- **실행 단계**: 메타데이터 추출 실행 → 작업 유형 분류 결과 확인.
- **기대 결과**: 정의된 집합 중 "버그 수정"으로 분류된다.
- **검증 AC**: AC1-1

### 시나리오 2: 작업 유형이 불명확한 PR
- **사전 조건**: 제목/설명/변경 내용만으로 유형 판단이 어려운 PR 입력.
- **실행 단계**: 분류 실행 → 결과 상태 확인.
- **기대 결과**: "미분류"로 표시되어 PRD-6 공백 탐지의 대상이 된다(오류로 끝나지 않는다).
- **검증 AC**: AC1-1

### 시나리오 3: 사람이 만든 PR 작성자 식별
- **사전 조건**: 일반 유저가 직접 생성한 PR.
- **실행 단계**: 작성자 식별 실행.
- **기대 결과**: 해당 유저가 그대로 작성자로 식별된다.
- **검증 AC**: AC1-2

### 시나리오 4: 봇 PR에서 실제 유저 추정
- **사전 조건**: 봇 계정이 생성한 PR(커밋 author·트리거 메타데이터에 실제 유저 흔적 존재).
- **실행 단계**: 작성자 식별 실행 → 추정 유저와 추정 근거 확인.
- **기대 결과**: 실제 유저가 추정되고 추정 근거(커밋 author 등)가 함께 기록된다. 추정 불가 시 "추정 불가"로 표시되어 공백으로 넘어간다.
- **검증 AC**: AC1-2

### 시나리오 5: 저장소 식별
- **사전 조건**: 특정 repository에서 생성된 PR.
- **실행 단계**: 저장소 식별 실행.
- **기대 결과**: 저장소 식별자가 정확히 추출되어 지식 베이스 저장소 키와 일치한다.
- **검증 AC**: AC1-3

### 시나리오 6: 동일 인물의 봇/직접 PR 키 정규화
- **사전 조건**: 같은 사람이 한 번은 직접, 한 번은 봇을 통해 생성한 두 PR.
- **실행 단계**: 두 PR의 작성자 키 정규화 결과 비교.
- **기대 결과**: 두 경우 모두 동일한 정규화 키로 통일되고, 키 형식이 지식 베이스 스키마와 일치한다.
- **검증 AC**: AC1-4

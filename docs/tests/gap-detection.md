# 테스트 문서: 논리 공백 탐지 및 보완

## 검증 대상 AC
- AC6-1: 비약/공백 탐지 (PRD-6)
- AC6-2: 보완 제안 생성 (PRD-6)
- AC6-3: 보완 후 재리뷰 (PRD-6)
- AC6-4: 리뷰 종결 (PRD-6)
- AC6-5: 코드베이스 변경 행위 금지 (PRD-6)

## 테스트 시나리오

### 시나리오 1: 근거 없는 주장 탐지
- **사전 조건**: 근거가 비어 있는 주장이 포함된 논리 체인.
- **실행 단계**: 공백 탐지 실행.
- **기대 결과**: 해당 주장이 공백으로 표시된다. 완결된 체인에서는 공백 0건이 보고된다.
- **검증 AC**: AC6-1

### 시나리오 2: 도약한 결론 탐지
- **사전 조건**: 선행 주장 없이 도출된 결론.
- **실행 단계**: 공백 탐지 실행.
- **기대 결과**: 결론이 비약으로 표시된다.
- **검증 AC**: AC6-1

### 시나리오 3: 공백별 보완 제안 생성
- **사전 조건**: 행위/정보/관점 부족으로 발생한 서로 다른 공백 3종.
- **실행 단계**: 보완 제안 생성 실행.
- **기대 결과**: 각 공백마다 "필요한 것: (행위|정보|관점) — 구체 내용" 제안이 1건 이상, 실행 가능한 형태로 생성된다.
- **검증 AC**: AC6-2

### 시나리오 4: 보완 입력 후 재리뷰
- **사전 조건**: 공백에 대해 사용자가 관련 이슈를 연결(보완)함.
- **실행 단계**: 보완 입력 → 재리뷰 트리거 → 결과 비교.
- **기대 결과**: 논리 엔진이 다시 돌아 결론이 갱신되고, 갱신 전후 차이가 추적 가능하다.
- **검증 AC**: AC6-3

### 시나리오 5: 종결 처리(공백 없음 / 사용자 감수)
- **사전 조건**: (a) 공백 0건 상태, (b) 공백이 남았지만 사용자가 종결 선택.
- **실행 단계**: 각 경우 종결 실행 → 상태/사유 확인.
- **기대 결과**: 종결 상태가 기록되고 사유(공백 없음 / 사용자 감수)가 남는다.
- **검증 AC**: AC6-4

### 시나리오 6: 코드 변경 행위 부재 확인 (경계)
- **사전 조건**: 종결 또는 재리뷰가 발생한 리뷰.
- **실행 단계**: 시스템 동작 집합과 PR 상태 변화 확인.
- **기대 결과**: 머지/승인/푸시 등 코드 변경 동작이 존재하지 않고, 종결/재리뷰가 PR의 머지 여부에 영향을 주지 않는다.
- **검증 AC**: AC6-5
